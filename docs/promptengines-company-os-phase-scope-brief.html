PromptEngines Operating Brief

Company OS Phase Outcomes & Scope

PromptEngines is a pseudo-open applied AI lab. It ships constantly, documents what works, pairs humans with agents, stays close to the work, and turns prototypes into products. This brief defines what success looks like after Phase 1 and Phase 2 of the Agent Org Chart Runtime + Company Operating System Agent program.

Current operating surface

Prompt Engines, Lab Notes, Consulting, Kaizen, Storybook Studio, Flow, Video Terminal, Norbu, and Dashboard.

Current team model

Andy Stable as human operator; A.I. as standing agent collaborator; Hermetic_Demiurge as an emerging named operator.

Intent

Make the company legible, controllable, and eventually compounding without turning the lab into bureaucracy.

Phase 1 — Company OS MVP

Goal: make PromptEngines legible, reviewable, and controllable from one operating surface.

Outcome after Phase 1

PromptEngines has a working Company OS MVP. Every major public node, agent, and core operating asset is registered, current state is visible in Dashboard, and the team can review, prioritize, approve, and ship from a shared control room.

How PromptEngines will be operating

  • Dashboard becomes the daily control room, not just another app.
  • The portfolio is managed as one operating surface instead of a loose set of repos, experiments, and docs.
  • Andy remains final approver on meaningful changes; agents gather state, draft actions, and prepare work for review.
  • A.I. and other named agents operate against shared registry data, clear ownership, and explicit approval gates.
  • Shipping and documentation stay coupled: Lab Notes and the build stream reflect real work, and real work is easier to recover and continue.

Phase 1 scope

  • Stand up the Company OS MVP: company registry, unified dashboard registry, event collector, dashboard control room, and daily operator loop with approval gates.
  • Register the current public surface area: Prompt Engines, Lab Notes, Consulting, Kaizen, Storybook Studio, Flow, Video Terminal, Norbu, and Dashboard.
  • Create one minimum shared operating model across products, agents, repos, docs, and domains: owner, status, health, next action, and approval state.
  • Capture enough operating signal to answer four daily questions: what exists, what changed, what needs attention, and what is ready to ship.
  • Make the human-plus-agent model explicit so agent activity accelerates operations without removing human judgment.

Phase 1 not in scope

  • Rebuilding every product or redesigning the full portfolio.
  • Full autonomy or unsupervised agent execution.
  • A heavyweight data platform, BI stack, or enterprise analytics program.
  • Deep workflow automation across every internal function.
  • Turning the Company OS into a public standalone product.
  • Expanding org complexity beyond the current small operator cell.

Phase 2 — Company OS as Operating Layer

Goal: make the operating layer useful enough to drive daily throughput across products, publishing, consulting, and internal ops.

Outcome after Phase 2

PromptEngines has a functioning human-agent operating system that not only sees the business but actively moves it forward. The company can route work, prepare outputs, manage bounded workflows, and keep product, publishing, consulting, and internal operations aligned through one layer.

How PromptEngines will be operating

  • The company runs from a shared operating layer: Dashboard for command, the registry for source of truth, and event flows for current state.
  • Agents do more than report; they reconcile state, draft deliverables, route follow-ups, and execute bounded workflows after approval.
  • Humans spend more time on direction, taste, prioritization, relationships, and final review; agents handle monitoring, synthesis, prep, and repeatable execution.
  • Public work and internal work reinforce each other: Lab Notes, consulting insight, product changes, and portfolio decisions stay connected instead of drifting apart.
  • PromptEngines behaves more like a coherent studio/lab with live portfolio management than a set of parallel experiments.

Phase 2 scope

  • Expand the registry and control room into a real operating layer across product ops, publishing, consulting, and internal admin.
  • Add bounded automations and agent workflows for monitoring, triage, drafting, publishing prep, routing, and follow-up.
  • Integrate core workspace and access flows — including team join, token management, and Google Workspace operational tooling — so identity and permissions support the human-agent model.
  • Make portfolio review a standing discipline: promote, pause, rebuild, or retire apps based on live signal and operator judgment.
  • Increase traceability of agent work, approvals, and decisions so the system remains steerable and auditable.
  • Strengthen the loop from prototype to product so promising nodes can move forward without adding process drag.

Phase 2 not in scope

  • Replacing founder/operator judgment with autonomous management.
  • End-to-end autonomous customer delivery or open-ended agent authority.
  • Building a generalized enterprise operating system for other companies.
  • Scaling low-signal products simply because they already exist in the portfolio.
  • Complex bespoke integrations where operating value is unclear.
  • Process layers that violate PromptEngines principles: ship constantly, document what works, stay close to the work, and just rebuild it when needed.

Strategic intent

Phase 1 makes PromptEngines legible and controllable. Phase 2 makes it compound. The goal is not more software for its own sake; it is a tighter human-agent studio that can ship more, learn faster, and manage a growing portfolio without losing founder judgment or lab speed.
